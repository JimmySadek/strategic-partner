## F1: Context Window Monitoring Is Unreliable

**Severity**: CRITICAL
**Audit Recommendation**: Hybrid approach — env var override baseline + optional companion script
**Files Changed**:
- `references/startup-checklist.md` — Step 3 adds `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=70`
- `references/context-handoff.md` — new "Environment Baseline" and "PreCompact Hook Integration" sections; updated thresholds
- `references/companion-script-spec.md` — NEW file, full architecture spec for external Python monitor
- `references/hooks-integration.md` — PreCompact hook configuration at 70%

**What Was Done**:
- Added env var `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=70` to startup sequence (Step 3)
- Integrated PreCompact hook as the authoritative backstop for context monitoring
- Updated handoff thresholds from 67/72/77% to 50-65/65-72/72%+ with strategic compaction in the middle tier
- Created companion-script-spec.md with full monitor architecture for power users
- Acknowledged self-assessment limitation explicitly in context-handoff.md

**Deviations from Audit**: None significant. The companion script remains a specification only (not implemented as a working Python script), which matches the audit's "advanced (for power users)" framing.

**Limitations**: The env var approach depends on Claude Code honoring `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`, which is a documented but relatively new feature. Self-assessment is still used for intermediate thresholds (50-65%, 65-72%).

**Commits**: `ad5003d`, `c22ecf8`, `861e7f5`, `2f8600d`

## F4: Prompt Crafting Quality — Core Deliverable at Risk

**Severity**: CRITICAL
**Audit Recommendation**: Mandatory parallelization check + routing decision tree + self-verification
**Files Changed**:
- `references/prompt-crafting-guide.md` — major update with three new mandatory sections
- `references/orchestration-playbook.md` — stronger parallelization heuristics with concrete examples

**What Was Done**:
- Added mandatory Routing Decision Tree (scope routing + complexity routing) that must be walked before writing any prompt
- Added mandatory 4-question Parallelization Check with quality gate: if Q1-3 answer YES and prompt lacks `<orchestration>`, the prompt FAILS
- Added 8-item Post-Craft Self-Verification checklist (mandatory, not advisory)
- Updated prompt quality requirement #1 to reference the decision tree explicitly
- Added 5 new anti-patterns: skipped parallelization check, missing orchestration when required, skipped routing tree, skipped post-craft verification, intuitive routing
- Orchestration playbook now includes concrete examples of WHEN to parallelize and when NOT to, plus model selection cost-effectiveness principle

**Deviations from Audit**: None. All three recommendations (A, B, C) implemented as specified.

**Post-Release Fix**: The routing decision tree initially used `(e.g., /gsd:quick)` and `(e.g., /gsd:debug)` annotations — concrete skill names that caused anchoring bias. This contradicted the v3.5.2 fix that deliberately removed hardcoded skill names. Fixed by replacing all concrete names with `(look up in routing matrix)` placeholders, restoring the v3.5.2 pattern. Anti-pattern warnings were also rewritten to avoid naming specific skills (negation-by-example reinforces the pattern it's trying to suppress).

**Limitations**: Quality gates are instruction-based, not mechanically enforced. The SP can still skip checks if instruction adherence degrades under context pressure — which is partly mitigated by the lean hub architecture (F11).

**Commits**: `f33ff6c`, `4d2e7ff`, `8f169f6`

## F6: `/compact` Ban Is Outdated

> ⚠️ **Later reversed (v4.0.2):** The strategic compact tier was removed — the SP
> now always hands off cleanly instead of suggesting /compact. Compaction produces
> lossy summaries and contradicts the fresh-session philosophy. See
> `references/context-handoff.md` § "Why the SP never suggests `/compact`" for the rationale.
>
> Keep all existing F6 content below the note as historical record. Do not delete anything.

**Severity**: HIGH
**Audit Recommendation**: Allow `/compact` with mandatory focus instructions
**Files Changed**:
- `references/partner-protocols.md` — new "/compact Protocol" section replacing the ban
- `references/context-handoff.md` — new "Strategic Compaction Protocol (65-72%)" section

**What Was Done**:
- Replaced the absolute `/compact` ban with a guardrailed protocol
- Bare `/compact` (no focus instructions) remains prohibited
- Strategic compaction with focus instructions allowed at 65-72% context
- Must be proposed via AskUserQuestion — never auto-compacted
- Added focus instruction template preserving: decisions, pending prompts, current goal, files modified, conventions
- One compaction per session max — if pressure returns, escalate to full handoff

**Deviations from Audit**: None. Implemented exactly as recommended.

**Limitations**: Focus instructions are advisory text — Claude Code's compaction algorithm may not perfectly preserve all specified content. Users should verify critical state survived after compaction.

**Commits**: `a962e47`, `c22ecf8`

## F9: Fire-and-Forget Agents May Silently Fail

**Severity**: CRITICAL
**Audit Recommendation**: Replace fire-and-forget with fire-and-verify
**Files Changed**:
- `references/startup-checklist.md` — Step 4 "Fire-and-Verify" agents, Step 6 verification gate
- `references/orchestration-playbook.md` — Pattern D updated to "Fire-and-Verify Operations"

**What Was Done**:
- Combined Serena dashboard fix and gitignore check into a single Agent C (fire-and-verify)
- Agent C returns structured results: `{ dashboard_fix, gitignore_fix }`
- Step 6 is a verification gate before orientation:
  - Gitignore failure → WARN USER IMMEDIATELY (security concern for shared/public repos)
  - Dashboard fix failure → note but don't block
- Pattern D in orchestration-playbook.md renamed from "Fire-and-Forget" to "Fire-and-Verify" with reference to startup-checklist.md for verification logic

**Deviations from Audit**: Combined the two agents into one (Agent C) rather than keeping them separate. This reduces agent overhead while maintaining verification.

**Limitations**: Verification requires the agent to return structured results. If the agent itself fails to spawn, the SP falls back to noting the limitation in orientation.

**Commits**: `ad5003d`, `4d2e7ff`

## F10: Routing Matrix Cost

**Severity**: MEDIUM
**Audit Recommendation**: Ship a curated base matrix with delta updates
**Files Changed**:
- `references/skill-routing-matrix.md` — major rewrite with curated base matrix and delta procedure

**What Was Done**:
- Added curated base matrix with ~30 pre-mapped skills across 7 categories (Implementation, Debugging, Quality, Architecture, Research, Documentation, Testing, Multi-Expert)
- Added delta-update procedure: load base -> scan for NEW skills -> build delta entries -> merge -> store in Serena
- On continuation: read cached matrix, diff against current skills, rebuild only if changed
- Estimated ~80% reduction in startup cognitive cost

**Deviations from Audit**: None. Implemented as recommended.

**Post-Release Fix**: The initial v4.0 implementation documented the delta-update procedure in `skill-routing-matrix.md` but **did not add a startup step that invokes it**. The v3.5.3 had "Skill + MCP inventory → routing matrix built → stored in Serena" as an explicit startup checklist item — this was lost during the restructure. Fixed by adding Step 5.5 to `startup-checklist.md` with the full delta-update flow: load base matrix → scan system context → build delta entries → merge with custom agents → store in Serena. The routing matrix is now a runtime artifact again, not just a static reference.

**Limitations**: The base matrix entries are curated for common skills. Unusual or niche skills still require delta building. The matrix needs periodic updates as new built-in agent types are added to Claude Code.

**Commits**: `d0d1824`, `8f169f6`

## F11: Skill Size and Modularization

**Severity**: MEDIUM
**Audit Recommendation**: Lean Hub + Reference Dispatch — slim SKILL.md to ~1,500 words
**Files Changed**:
- `SKILL.md` — major restructuring from ~700 lines to ~257 lines

**What Was Done**:
- Restructured SKILL.md as a lean hub with core behaviors inline and reference dispatch:
  1. Persona identity and implementation firewall (always present)
  2. Mode detection with pointer to startup-checklist.md
  3. All 7 responsibilities with substance (not just pointers) — including Serena edge cases table, Git Custody with post-impl verification, Self-Delegation Principle
  4. Routing table for when to load each reference file
- Moved procedural content (startup steps, handoff mechanics, hook configs) to reference files
- Added 2 new reference files to the routing table: hooks-integration.md, companion-script-spec.md
- Each reference file entry now has a clear "When to Load" trigger
- Version bumped to 4.0.0 in frontmatter

**Post-Release Fix**: Initial restructure was too aggressive (~256 lines) and lost core SP behaviors. Gap analysis against v3.5.3 identified missing: Self-Delegation Principle, 10-point prompt quality list, ══ fence format, Ask-Before-Act examples, Communication Style details. Restored to ~440 lines — still a 41% reduction from v3.5.3's ~700 lines, but now retains all core behaviors.

**Deviations from Audit**: The final size (~440 lines / ~2,566 words) is above the audit's ~1,500 word recommendation but this was a deliberate decision after user feedback that core behaviors must have substance in SKILL.md, not just reference pointers. The principle adopted: keep WHAT and WHY in core, extract only HOW to references.

**Limitations**: Reference files must be loaded on demand, which means the SP needs to correctly identify when each reference is needed. If it fails to load the right reference, behavior may degrade.

**Commits**: `62bf02c`
